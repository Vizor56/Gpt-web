# Полное описание проекта онлайн-школы

Документ описывает текущее устройство проекта, основные роли, функции сайта, базу данных, серверную логику, деплой и тестовые аккаунты. Он нужен как карта проекта для разработки, проверки и дальнейшего развития.

## 1. Назначение проекта

Проект — сайт онлайн-школы подготовки к ЕГЭ/ОГЭ с личными кабинетами для учеников, преподавателей, кураторов и администраторов.

Сайт решает несколько задач:

- показывает ученикам курсы, уроки, конспекты, домашние задания, трансляции и созвоны;
- принимает заявки с главной страницы и обращения в поддержку;
- даёт ученикам возможность сдавать домашнее задание ссылкой на Google Drive;
- позволяет преподавателям проверять ДЗ, ставить оценку от 1 до 10 и писать комментарий;
- позволяет кураторам оценивать материалы преподавателей и видеть прогресс проверки;
- позволяет администраторам управлять заявками, учениками, родителями, преподавателями, кураторами, курсами и связями между ними;
- хранит данные в PostgreSQL Neon;
- деплоится на Render как Node.js Web Service.

## 2. Технологии

Основной стек:

- Frontend: обычные `index.html`, `styles.css`, `script.js` без сборщика.
- Backend: Node.js 20+, Express.
- Database: PostgreSQL, рассчитано на Neon.
- Hosting: Render Web Service.
- Работа с базой: пакет `pg`.
- Конфигурация окружения: `dotenv`.

Ключевые файлы:

- `Сайт с помощью гпт/index.html` — разметка сайта и SVG-иконки.
- `Сайт с помощью гпт/styles.css` — весь дизайн, адаптивность, интерфейсы ролей, магазин и модальные окна.
- `Сайт с помощью гпт/script.js` — клиентская логика, навигация, запросы к API, отрисовка всех кабинетов.
- `Сайт с помощью гпт/server.mjs` — Express-сервер, API, авторизация, работа с PostgreSQL, seed-данные.
- `Сайт с помощью гпт/database/neon-schema.sql` — основная схема базы данных.
- `Сайт с помощью гпт/package.json` — команды запуска для Render.
- `render.yaml` и `Сайт с помощью гпт/render.yaml` — конфигурация Render.

## 3. Запуск и деплой

### Локальный запуск

Нужен Node.js 20+.

```bash
cd "Сайт с помощью гпт"
npm install
npm start
```

По умолчанию сервер запускается на:

```text
http://127.0.0.1:8765
```

Если открыть `index.html` напрямую через файл, frontend всё равно пытается обращаться к API по адресу `http://127.0.0.1:8765`. Поэтому для полной работы функций нужен запущенный сервер.

### Деплой на Render

В Render нужно создать Web Service из GitHub-репозитория.

Если Render настроен на подпапку проекта:

```text
Root Directory: Сайт с помощью гпт
Build Command: npm install
Start Command: npm start
```

Переменные окружения:

```text
DATABASE_URL=postgresql://...sslmode=require
NODE_ENV=production
AUTO_MIGRATE=true
```

`DATABASE_URL` берётся из Neon. Секреты нельзя коммитить в GitHub.

### Автомиграции

При старте сервер:

1. читает `DATABASE_URL`;
2. подключается к PostgreSQL;
3. если `AUTO_MIGRATE` не равен `false`, применяет `database/neon-schema.sql`;
4. выполняет runtime-миграции из `server.mjs`;
5. добавляет стартовые данные: курсы, уроки, преподавателей, кураторов, администраторов, ученика, ДЗ, конспекты, трансляции, магазин.

## 4. Роли пользователей

В системе есть четыре типа пользователей.

### Ученик

Ученик видит обычную версию сайта:

- главная страница;
- учеба;
- сообщения;
- баллы;
- библиотека конспектов;
- поддержка;
- аккаунт;
- магазин за баллы.

До входа в аккаунт:

- личные баллы и ФИО скрыты;
- в каждом курсе доступен первый урок;
- конспекты доступны только для первых уроков;
- остальные материалы закрыты.

После входа:

- сайт показывает купленные курсы;
- подтягивает прогресс, баллы, домашние задания, трансляции, созвоны;
- открывает материалы купленных курсов;
- даёт сдавать ДЗ ссылкой на Google Drive.

### Преподаватель

После входа преподаватель попадает в отдельную версию сайта без обычной главной страницы ученика.

У преподавателя есть разделы:

- сообщения;
- баллы;
- ученики;
- домашние задания;
- курсы;
- трансляции.

Преподаватель может:

- видеть прикреплённых учеников;
- видеть курсы, которые он ведёт;
- открывать уроки по курсам;
- создавать и редактировать уроки;
- добавлять ссылку на видеозапись, конспект и ДЗ;
- автоматически создавать связанные конспекты и шаблоны ДЗ при создании урока;
- создавать и редактировать трансляции;
- проверять ДЗ учеников;
- ставить оценку от 1 до 10;
- писать комментарий ученику;
- видеть архив уже проверенных ДЗ;
- писать сообщения ученикам, администраторам и доступным участникам чатов.

### Куратор

Куратор тоже видит отдельную версию сайта.

У куратора есть разделы:

- сообщения;
- баллы;
- преподаватели;
- ученики;
- уроки;
- конспекты;
- домашние задания;
- трансляции;
- созвоны;
- архив оценённых материалов.

Куратор может:

- видеть преподавателей, закреплённых за ним;
- видеть учеников этих преподавателей;
- открывать карточку преподавателя и видеть его курсы и учеников;
- оценивать уроки, конспекты, ДЗ и трансляции преподавателя по шкале 1-10;
- писать комментарии к материалам;
- видеть архив уже оценённых материалов;
- создавать и редактировать созвоны с учениками;
- выбирать учеников-участников созвона;
- оставлять комментарии к ученикам, видимые преподавателям и администраторам;
- видеть статистику ДЗ по урокам: всего учеников, сколько сдали, сколько проверено преподавателем, сколько оценено куратором.

### Администратор

Администратор входит через общий вход команды, но попадает в отдельную админскую версию сайта.

Администратор видит разделы:

- заявки;
- обращения;
- ученики;
- родители;
- преподаватели;
- кураторы;
- сообщения;
- уведомления.

Администратор может:

- видеть заявки с главной страницы;
- менять данные заявки;
- помечать заявку как обработанную;
- видеть архив обработанных заявок;
- видеть обращения в поддержку;
- менять статус обращения;
- добавлять и редактировать учеников;
- создавать ученику логин и пароль;
- открывать и закрывать ученику доступ к курсам;
- менять преподавателя и куратора ученика;
- ставить статус ученика по курсам;
- добавлять и редактировать родителей;
- связывать родителей с детьми;
- добавлять и редактировать преподавателей;
- выбирать курсы преподавателя;
- видеть учеников преподавателя;
- менять куратора преподавателя;
- добавлять и редактировать кураторов;
- выбирать преподавателей, закреплённых за куратором;
- создавать чаты с учениками, преподавателями и кураторами;
- сортировать чаты по типу собеседника.

## 5. Основные разделы ученического сайта

### Главная страница

Главная страница содержит:

- краткую информацию о школе;
- блок курсов;
- форму заявки;
- ближайшие трансляции и созвоны;
- переходы в учебные разделы.

Форма заявки реально отправляется в базу через API. После отправки заявка появляется у администратора в разделе заявок со статусом необработанной.

### Учёба

В разделе учебы есть вкладки:

- купленные курсы;
- все курсы.

Купленные курсы помечаются отдельно. Для некупленных курсов ученику доступен первый урок как демо-доступ.

В карточке курса показывается:

- название курса;
- количество уроков;
- количество ДЗ;
- прогресс;
- переходы к урокам, ДЗ, конспектам и трансляциям.

Прогресс курса считается по проверенным ДЗ:

```text
процент курса = проверенные ДЗ / все назначенные ДЗ * 100
```

### Страница курса

Внутри курса есть вкладки:

- уроки;
- ДЗ;
- конспекты;
- трансляции.

Во вкладке уроков отображаются только уроки. Для каждого урока работают кнопки:

- запись;
- конспект;
- ДЗ.

Для ДЗ используется цветовая логика:

- жёлтый — ДЗ не сдано;
- синий — ДЗ отправлено на проверку;
- зелёный — ДЗ проверено.

Если ДЗ проверено, рядом показывается оценка, например:

```text
9 / 10 баллов
```

При открытии проверенного ДЗ ученик видит:

- статус проверки;
- оценку;
- комментарий преподавателя;
- ссылку на задание;
- ссылку на свою работу;
- кнопку копирования ссылки.

### Сдача домашнего задания

Ученик не загружает файл на сервер. Он вставляет ссылку на файл/документ в Google Drive.

Логика:

1. ученик открывает ДЗ;
2. вставляет ссылку Google Drive;
3. сайт отправляет ссылку в API;
4. сервер сохраняет ссылку в `homework_submissions`;
5. статус ДЗ становится `Submitted`;
6. преподаватель получает уведомление о сдаче;
7. преподаватель проверяет ДЗ;
8. статус становится `Checked`;
9. ученик получает уведомление о проверке и видит оценку/комментарий.

Сервер запрещает менять ссылку, если ДЗ уже проверено.

### Библиотека конспектов

Библиотека конспектов подтягивает материалы из базы.

До покупки курса:

- доступны только конспекты первых уроков.

После покупки курса:

- доступны конспекты всех уроков купленного курса.

### Трансляции и созвоны

Ближайшие события подтягиваются из базы.

В блоке ближайшего показываются:

- будущие трансляции;
- будущие созвоны с куратором.

События с прошедшей датой не отображаются.

Домашние задания в блоке ближайшего не показываются.

### Поддержка

Страница поддержки содержит форму обращения. Обращение отправляется в базу и появляется у администратора в разделе обращений.

### Магазин за баллы

У ученика есть магазин, где можно тратить баллы.

Возможности:

- покупать предметы;
- надевать и снимать предметы;
- менять 2D-питомца;
- менять цвет шерсти и глаз;
- менять окружение;
- видеть выбранного питомца как аватар аккаунта.

Баллы начисляются за сдачу ДЗ. История начислений хранится в `point_transactions`.

## 6. Уведомления

Уведомления открываются по колокольчику рядом с аккаунтом. Счётчик сбрасывается после открытия уведомлений.

Ученик получает уведомления:

- о новой трансляции;
- о проверенном ДЗ с предметом, уроком и оценкой.

Преподаватель получает уведомления:

- какой ученик сдал какое ДЗ;
- по какому предмету и уроку;
- во сколько отправлено;
- сколько ДЗ сдано из возможных;
- сколько из них проверено.

Куратор получает уведомления:

- какой преподаватель проверил какое ДЗ;
- по какому предмету и уроку;
- сколько ДЗ сдано;
- сколько проверено преподавателем;
- сколько получило фидбек куратора.

Администратор получает уведомления:

- о новых заявках;
- о новых обращениях;
- сколько заявок необработано;
- как давно пришла самая старая необработанная заявка.

## 7. Сообщения и чаты

Чаты хранятся в таблицах `chats` и `chat_messages`.

Поддерживаются разные типы чатов:

- ученик-преподаватель;
- ученик-куратор;
- преподаватель-администратор;
- куратор-администратор;
- администратор-ученик.

Чаты не создаются все сразу для всех возможных связей. Пользователь выбирает, кому написать, после чего сайт создаёт или открывает соответствующий чат.

Администратор может сортировать чаты:

- с учениками;
- с преподавателями;
- с кураторами.

## 8. Домашние задания и проверка

Домашнее задание состоит из нескольких сущностей:

- `homework_templates` — шаблон ДЗ, связанный с уроком и курсом;
- `homework_assignments` — назначенное ДЗ конкретному ученику;
- `homework_submissions` — сдача ДЗ учеником;
- `point_transactions` — начисление баллов ученику.

Статусы:

- `Assigned` — ДЗ назначено, но не сдано;
- `Submitted` — ученик отправил ссылку, работа на проверке;
- `Checked` — преподаватель проверил работу;
- `Cancelled` — ДЗ отменено.

Оценки:

- все оценки ставятся по шкале от 1 до 10;
- преподаватель ставит оценку ученику за ДЗ;
- куратор ставит оценки материалам преподавателя;
- рейтинг преподавателя считается по оценкам кураторов.

Преподаватель после проверки не видит форму повторной оценки той же работы как активную: вместо неё отображается результат проверки. У куратора оценённые материалы уходят в архив.

## 9. Навигация преподавателя и куратора по ДЗ

Для преподавателя и куратора ДЗ организованы по шагам:

1. список курсов;
2. список уроков выбранного курса;
3. страница конкретного урока с работами учеников.

Рядом с уроками показывается статистика:

- всего ДЗ/учеников;
- сколько сдано;
- сколько проверено преподавателем;
- для куратора — сколько оценено куратором.

Цветовая логика:

- у преподавателя урок подсвечивается зелёным, если все сданные ДЗ проверены;
- у куратора урок подсвечивается зелёным, если все проверенные преподавателем ДЗ получили фидбек куратора.

## 10. Управление курсами и материалами

Преподаватель может создавать уроки в своих курсах.

При создании урока можно указать:

- название;
- тему;
- ссылку на видео;
- ссылку на конспект;
- ссылку на ДЗ;
- дату и время трансляции.

Если указана ссылка на конспект, создаётся запись в `notes`.

Если указана ссылка на ДЗ, создаётся запись в `homework_templates`, а ученикам курса назначаются связанные ДЗ в `homework_assignments`.

Трансляции хранятся в `live_streams` и подтягиваются в ученический, преподавательский и кураторский интерфейсы.

## 11. Администрирование связей

Администратор управляет связями между сущностями:

- ученик — курсы;
- ученик — преподаватель;
- ученик — куратор;
- родитель — дети;
- преподаватель — курсы;
- преподаватель — куратор;
- куратор — преподаватели.

Эти связи хранятся в базе и используются всеми версиями интерфейса. Например, если администратор прикрепляет ученика к преподавателю, этот ученик появляется в кабинете преподавателя.

## 13. Авторизация

Авторизация реализована внутри проекта без внешнего сервиса.

Пароли хранятся как SHA-256-хеши.

После входа сервер создаёт `auth_token` и срок действия сессии. Клиент отправляет токены в заголовках:

- для ученика: `X-Student-Id`, `X-Auth-Token`;
- для команды: `X-Staff-Id`, `X-Staff-Role`, `X-Staff-Auth-Token`.

Роли команды:

- `Teacher`;
- `Curator`;
- `Admin`.

## 14. Основные API

Ученические API:

- `POST /api/auth/register` — регистрация ученика;
- `POST /api/auth/login` — вход ученика;
- `GET /api/account` — данные аккаунта ученика;
- `GET /api/courses/:slug/lessons` — уроки курса;
- `GET /api/homeworks` — ДЗ ученика;
- `POST /api/homeworks/:id/submit` — отправка ссылки на ДЗ;
- `GET /api/streams` — трансляции;
- `GET /api/student/calls` — созвоны ученика;
- `POST /api/applications` — заявка с главной;
- `POST /api/support` — обращение в поддержку;
- `POST /api/shop/purchase` — покупка товара;
- `POST /api/shop/equip` — надеть/снять товар.

Командные API:

- `POST /api/staff/login` — вход преподавателя, куратора или администратора;

## 16. Как добавить администратора вручную в Neon

В Neon Console можно выполнить:

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;

INSERT INTO admin_accounts (first_name, last_name, login, password_hash)
VALUES (
  'Имя',
  'Фамилия',
  'new_admin',
  encode(digest('new_password123', 'sha256'), 'hex')
);
```

После этого вход:

```text
new_admin / new_password123
```

## 17. Что важно помнить при разработке

- Не коммитить `.env` и реальные секреты.
- После изменений в `server.mjs` проверять синтаксис Node.
- После изменений в `script.js` проверять сайт в браузере, потому что весь frontend живёт в одном большом файле.
- При изменении схемы базы обновлять `database/neon-schema.sql` и, если нужно, runtime-миграции в `server.mjs`.
- При изменении логики ролей проверять все роли: ученик, преподаватель, куратор, администратор.
- Для Render важно, чтобы `DATABASE_URL`, `NODE_ENV=production`, `AUTO_MIGRATE=true` были заданы в окружении.

## 18. Текущее состояние проекта

На данный момент базовый макет школы реализован и включает:

- полноценный ученический интерфейс;
- отдельный интерфейс преподавателя;
- отдельный интерфейс куратора;
- отдельный интерфейс администратора;
- рабочую регистрацию и вход;
- курсы, уроки, конспекты, ДЗ, трансляции и созвоны;
- сдачу ДЗ ссылкой на Google Drive;
- проверку ДЗ с оценкой и комментарием;
- прогресс курсов по проверенным ДЗ;
- баллы и магазин с 2D-питомцем;
- заявки и обращения;
- чаты и уведомления;
- хранение данных в Neon PostgreSQL;
- деплой на Render.

Проект остаётся простым по инфраструктуре: без React, без сборщика, без отдельного ORM. Главный плюс такого подхода — его легко деплоить и быстро менять. Главный минус — `script.js` и `server.mjs` уже большие, поэтому при дальнейшем росте проекта стоит постепенно делить код на модули.
